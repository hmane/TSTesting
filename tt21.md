# AI Tagging — Fields and Approach

---

## Fields in scope for AI auto-tagging

| Field | Type | AI source |
|---|---|---|
| **Subject Matter** | Multi-select | OpenAI classification |
| **Contract Type** | Single-select | OpenAI classification |
| **Term Category** | Single-select | OpenAI reasoning |
| **Effective Date** | Date | OpenAI reasoning |
| **End Date** | Date | OpenAI reasoning |
| **Action Date Notice Period** | Text | OpenAI extraction |
| **Contract Description/Notes** | Free text | OpenAI summarization |

### Out of scope for AI

- **Owning Business Unit** — autofilled from data entered at request creation
- **Business Owners** — autofilled from data entered at request creation
- **Status** — workflow-managed
- **Signed** — boolean, not extracted

---

## Two approaches: short term vs long term

### Short term — Analyst-triggered AI Assist (Phase 1)

#### What

A new **AI Assist** button is added to the existing Request form. When the analyst is ready to tag, they click the button. The form fields are pre-populated with AI suggestions, each showing a confidence indicator (High / Medium / Low) and a tooltip explaining where the AI found each value in the contract. Analyst reviews, edits if needed, and saves.

#### Flow

1. Analyst opens the executed contract in the existing form
2. Clicks **AI Assist**
3. SPFx calls an Azure Function
4. Function fetches the PDF from SharePoint
5. Function calls Document Intelligence (`prebuilt-contract`) → structured facts (parties, execution date, jurisdictions)
6. Function calls Azure OpenAI in two passes:
   - **Pass 1 — Classification:** determines Contract Type and Subject Matter
   - **Pass 2 — Extraction:** uses a contract-type-specific prompt to extract Term Category, Effective Date, End Date, Action Date Notice Period, Contract Description
7. Function returns suggestions with confidence and rationale per field
8. Form pre-populates fields with AI values + confidence badges
9. Analyst reviews, edits, saves
10. Both AI suggestion and final analyst value are stored — enables audit and accuracy measurement

#### Why two OpenAI passes instead of one

- A contract-type-specific prompt (e.g., an NDA prompt) is more accurate than a generic prompt covering all types. The NDA prompt knows what to look for and what *not* to look for. A generic prompt is necessarily weaker on each type.
- Fields like End Date and Action Date Notice Period have interdependencies that vary by contract type. Type-aware extraction produces more internally consistent results.
- Easier to maintain — improving extraction for one contract type doesn't risk breaking another.
- The cost of the extra call is minimal compared to the accuracy gain.

#### Why this approach for the short term

- **Analyst stays in control** — every value is reviewed and confirmed; zero business risk from wrong AI suggestions
- **No custom AI training needed** — uses prebuilt models that handle all three of our template scenarios (our template, our template with vendor edits, vendor's own template)
- **Builds the data foundation** — accuracy measurement from day one drives long-term decisions
- **Existing workflow preserved** — AI is additive, not replacing anything; if AI fails, analyst tags manually as today
- **Builds analyst trust progressively** — suggestions earn credibility before any automation removes them from the loop

#### What we get

- Significant reduction in analyst tagging time, especially on our-template contracts
- Better consistency across analysts (everyone starts from the same AI baseline)
- Per-field accuracy data to inform long-term decisions
- Reusable AI orchestration pattern for future document workflows

### Long term — Background processing + selective auto-acceptance (Phases 2–3)

#### What

**Phase 2: Background processing.** AI tagging runs automatically the moment a contract is executed (signed). A SharePoint webhook drops a message on a queue, the same Azure Function processes it in the background, and Power Automate notifies the analyst when suggestions are ready. The analyst becomes a reviewer rather than a starter.

**Phase 3: Selective auto-acceptance.** For high-confidence fields on contracts using our template, auto-accept the AI value without analyst review. Analysts only see contracts that need attention — vendor templates, low-confidence fields, edge cases.

#### Flow

1. Contract is executed and uploaded to Request Documents
2. SharePoint webhook fires → message dropped on Azure Storage Queue
3. Same Azure Function processes asynchronously (same code path as short term, different trigger)
4. AI suggestions written back to the SharePoint list item
5. **Phase 2:** Power Automate sends analyst a Teams/email notification → analyst opens form (already pre-populated) → reviews and saves
6. **Phase 3:** For fields where confidence is above threshold AND historical accuracy is consistently high, value is auto-accepted and contract auto-promoted to Contract Library. Analyst only notified if confidence is low or template is unfamiliar.

#### Why this approach for the long term

- **Captures the full efficiency win** — suggestions are ready before the analyst opens the contract
- **Phase 3 shifts the model** — analyst becomes an exception handler instead of tagging every contract; throughput scales without headcount
- **Reuses Phase 1 infrastructure** — same Function, same AI services, only the trigger and the level of automation change
- **Confidence-based safety** — auto-acceptance only enabled where accuracy is proven; everything else still routes through analyst review
- **Earned trust** — only viable after Phase 1 produces real accuracy data showing the AI is reliable

#### What we get

- **Phase 2:** Suggestions ready before the analyst opens the contract; analyst becomes a reviewer
- **Phase 3:** Auto-tagged contracts move straight to Contract Library; analyst time refocused on complex contracts and exceptions
- **Compounding value** — Phase 1 reduces *time per contract*; Phase 3 reduces *number of contracts needing human attention at all*

---

## Why phased, not all-at-once

- **De-risks** — each phase validates the next; problems are found before automation hides them
- **Builds analyst trust** — automation comes after suggestions have earned credibility, not before
- **Data-driven decisions** — Phase 3 auto-acceptance is justified by real accuracy data, not estimates
- **Existing workflow preserved at every stage** — manual tagging remains functional; AI outages don't block the business

## Summary

| | Short term (Phase 1) | Long term (Phases 2–3) |
|---|---|---|
| **Trigger** | Analyst clicks "AI Assist" | Automatic on contract execution |
| **Analyst role** | Reviews every contract | Reviews exceptions only |
| **Automation** | Human-in-the-loop | Selective auto-acceptance |
| **Business risk** | None — analyst confirms everything | Low — only auto-accept where accuracy is proven |
| **Primary value** | Reduces time per contract | Reduces contracts needing human attention |
