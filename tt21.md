# AI Tagging — Fields and Approach

---

## Fields in scope for AI auto-tagging

| Field | Type | AI source |
|---|---|---|
| **Subject Matter** | Multi-select | Azure OpenAI classification |
| **Contract Type** | Single-select | Azure OpenAI classification |
| **Term Category** | Single-select | Azure OpenAI reasoning |
| **Effective Date** | Date | Azure OpenAI reasoning |
| **End Date** | Date | Azure OpenAI reasoning |
| **Action Date Notice Period** | Text | Azure OpenAI extraction |
| **Contract Description/Notes** | Free text | Azure OpenAI summarization |

**Out of scope for AI:** Owning Business Unit and Business Owners (autofilled from request creation), Status (workflow-managed), Signed (boolean).

### Classification vs reasoning

The AI is doing two different things across these fields:

- **Classification** — pick from a known list (Subject Matter, Contract Type). Answer space is closed.
- **Reasoning** — read the text and work out the answer (Term Category, dates, notice period). Answer space is open; sometimes requires computation ("three years from Effective Date" → End Date).

Both return value + confidence + rationale. They fail differently: classification errors are wrong-category (easy for the analyst to spot), reasoning errors are wrong-value (which is why the rationale always cites the contract section so the analyst can verify the derivation, not just the result). Classification fields will typically qualify for Phase 3 auto-acceptance before reasoning fields, because bounded answer spaces are easier to validate.

## Short term — Analyst-triggered AI Assist

A new **AI Assist** button is added to the existing Request form. The analyst clicks it when ready to tag; the form pre-populates with AI suggestions plus confidence indicators and rationale tooltips; analyst reviews, edits, saves. Existing workflow is preserved as a fallback if AI fails.

### Flow

1. Analyst clicks **AI Assist** in the form
2. SPFx calls Azure Function with request and document IDs
3. Function fetches the PDF from SharePoint
4. Function calls **Document Intelligence (`prebuilt-contract`)** → structured facts (parties, execution date, jurisdictions)
5. Function calls **Azure OpenAI** in two passes:
   - **Pass 1** classifies Contract Type and Subject Matter
   - **Pass 2** uses a type-specific prompt to extract Term Category, dates, notice period, description
6. Form renders suggestions with confidence badges (High/Medium/Low) and rationale tooltips
7. Analyst reviews, edits, saves. Both AI suggestion and final value are stored for audit and accuracy measurement. Each processing run is correlated with the request and document IDs for traceability across SharePoint, Azure Function, Document Intelligence, and Azure OpenAI calls.

> **Why Azure Function (not Power Automate)?** The pipeline needs prompt routing, retries, chunking, and structured error handling — better suited to code than declarative flows. Power Automate is well-suited for notifications, which is where it's used in Phase 2.

### What each stage sends and receives

**Stage 1 — Function → Document Intelligence**

- **Sent:** the PDF (binary). No prompt.
- **Returned:** structured extraction with confidence and page locations.

```json
{
  "parties": [
    { "name": "Acme Industries, Inc.", "role": "Provider", "confidence": 0.99 },
    { "name": "Contoso Ltd.", "role": "Customer", "confidence": 0.99 }
  ],
  "executionDate": { "value": "2026-03-15", "confidence": 0.98 },
  "jurisdiction": { "value": "California", "confidence": 0.97 },
  "sections": [
    { "title": "Term",        "pageRange": [3, 4] },
    { "title": "Termination", "pageRange": [4, 5] },
    { "title": "Renewal",     "pageRange": [5, 5] },
    { "title": "Notice",      "pageRange": [5, 6] }
  ]
}
```

DI returns facts and section locations only — no business categories, no interpretation. The section locations enable targeted clause retrieval and chunking for downstream extraction stages. Where section identification is incomplete or inconsistent (vendor templates with custom headings, clauses buried in appendices), the Function falls back to heuristic chunking and broader context extraction.

**Stage 2 — Function → Azure OpenAI Pass 1 (classification)**

- **Sent:**
  - The DI structured extraction (above)
  - First 2–3 pages of contract text + the signature page (usually enough to classify)
  - Classification prompt containing the full list of allowed Contract Types and Subject Matter values
- **Returned:**

```json
{
  "contractType": { "value": "NDA/Confidentiality Agreement", "confidence": 0.97 },
  "subjectMatter": { "value": ["Professional Services"], "confidence": 0.82 }
}
```

This result tells the Function which type-specific prompt to load for Stage 3.

**Stage 3 — Function → Azure OpenAI Pass 2 (reasoning extraction)**

- **Sent:**
  - The DI structured extraction (so Azure OpenAI has the parties and execution date as reference points)
  - Stage 2's Contract Type (so the right prompt is selected)
  - Only the relevant contract sections — Term, Termination, Renewal, Notice — pulled from the page ranges DI identified in Stage 1
  - Type-specific extraction prompt (e.g., the NDA prompt) with the JSON schema for the output
- **Returned:** the remaining fields with value + confidence + rationale per field (matches the Sample output earlier in this document)

The Function then merges Stage 1's facts, Stage 2's classifications, and Stage 3's extractions into the single response the form receives.

### Why Document Intelligence first

- **Prebuilt by Microsoft, no training needed** — accurate on contract facts (parties, dates, jurisdictions) out of the box
- **Handles all three template scenarios** — looks for contract concepts, not fixed layouts
- **Faster and cheaper than Azure OpenAI** for what it does — establishes a reliable baseline before interpretation
- **Schema is fixed, no reasoning** — doesn't extract business-specific fields like Term Category or compute Effective Date from clauses, which is why Azure OpenAI runs after

### Why Azure OpenAI in two passes

- **Covers what DI can't** — Term Category, computed dates (e.g., "effective 90 days after execution"), notice periods, summaries
- **DI extraction constrains the LLM** — structured input reduces unsupported inferences and improves consistency
- **Type-specific prompts are more accurate** — an NDA prompt knows what NDAs have and don't have; a one-prompt-fits-all is necessarily weaker
- **Pass 1 determines the extraction strategy for Pass 2** — once Contract Type is detected, the right prompt template runs for extraction
- **Cost of the extra call is minimal vs the accuracy gain**

### Why not just send the whole PDF to Azure OpenAI?

A fair question — modern LLMs are technically capable of processing 40+ page documents. We don't take that approach because:

- **Cost** — a 40-page contract is roughly 25,000–40,000 input tokens; sending it to Azure OpenAI twice (Pass 1 + Pass 2) adds up fast. DI runs once at flat per-page cost, and Pass 2 only sees a few thousand tokens of relevant clauses.
- **Accuracy** — counterintuitively, more context isn't better. LLMs get distracted by irrelevant sections (definitions, exhibits, signature blocks); focused chunks of relevant text consistently outperform full-document context for targeted extraction.
- **Context window limits** — long MSAs with schedules and exhibits can exceed even GPT-4's context window; without chunking, those contracts can't be processed at all.
- **No structured anchors** — DI gives us validated facts (parties, execution date) that anchor Azure OpenAI's reasoning; without them, Azure OpenAI interprets raw text with no ground truth to cross-check, raising hallucination risk.
- **Harder to debug** — DI's extraction is deterministic and easy to validate. Separating the deterministic step from the probabilistic step means when a value is wrong, we can tell whether the issue is parsing or interpretation.

The two-service approach trades small added complexity for meaningfully better cost, accuracy, and operability.

    },
    "EffectiveDate": {

    },
    "EndDate": {
      "value": "2029-03-14",

    },
    "TermCategory": {
      "value": "Expiration",

Each field returns **value + confidence + rationale**. The form uses confidence for the badge and shows the rationale as a tooltip so the analyst can verify each suggestion without re-reading the contract. Numeric confidence and supporting quality signals (model score, OCR quality, clause consistency, template familiarity) are stored for telemetry and threshold logic; the form renders banded badges only, to avoid implying false precision.

### Edge cases and abstention

The system is designed to **abstain rather than generate unsupported values** when evidence is weak:

- **Insufficient or conflicting evidence** — field returns no suggestion rather than a low-confidence value; conflicting clauses (e.g., amendment superseding original term) are flagged with both interpretations in the rationale
- **Scanned or low-quality PDFs** — OCR confidence is checked first; below threshold, the contract routes to manual review and no AI suggestions are generated
- **Long contracts** — Function preprocesses into logical sections (term, termination, notice, renewal) and sends only relevant chunks to Azure OpenAI, keeping cost and accuracy manageable
- **Legally ambiguous fields** — especially End Date and Term Category with auto-renewals, conditional renewals, or survival clauses; AI provides a *recommendation with rationale*, the determination remains the analyst's
- **Transient failures** — Azure OpenAI throttling, OCR timeout, or network interruption are retried automatically with exponential backoff; unrecoverable failures route the contract to manual processing without blocking workflow progression

A "no suggestion" outcome is treated as a valid result. We prefer it over a confident-looking wrong answer.

## Long term — Background processing + selective auto-acceptance

Same architecture, different trigger and level of automation.

**Phase 2 — Background processing.** A SharePoint webhook fires when a contract is executed, drops a message on a queue, the same Function processes it asynchronously, Power Automate notifies the analyst when suggestions are ready. Analyst becomes a reviewer rather than a starter.

**Phase 3 — Selective auto-acceptance.** For high-confidence fields on contracts using our template, auto-accept the AI value *per field* without analyst review. Auto-acceptance is field-by-field — Contract Type or Subject Matter may qualify before End Date or Notice Period. Analysts only see contracts that have low-confidence fields, unfamiliar templates, or other exceptions.

### Why this for the long term

- **Suggestions ready before analyst opens the contract** (Phase 2)
- **Auto-acceptance only enabled where accuracy is proven** (Phase 3) — confidence threshold + historical accuracy data from Phase 1 gate which fields can auto-accept
- **Reuses Phase 1 infrastructure** — same Function, same AI services, only the trigger changes
- **Compounding value** — Phase 1 reduces *time per contract*; Phase 3 reduces *number of contracts needing human attention*

---

## Why phased, not all at once

- **De-risks** — each phase validates the next; accuracy proven before automation removes the human
- **Builds analyst trust** — automation comes after suggestions have earned credibility
- **Data-driven** — Phase 3 auto-acceptance is justified by real accuracy data, not estimates
- **Existing workflow preserved** — manual tagging remains functional; AI failures don't block the business

---

## Summary

| | Short term | Long term |
|---|---|---|
| **Trigger** | Analyst clicks "AI Assist" | Automatic on contract execution |
| **Analyst role** | Reviews every contract | Reviews exceptions only |
| **AI role** | Suggestion engine | Exception-focused automation |
| **Automation** | Human-in-the-loop | Selective auto-acceptance |
| **Business risk** | Low — analyst confirms all values | Low — only auto-accept where accuracy is proven |
| **Primary value** | Reduces time per contract | Reduces contracts needing human attention |
