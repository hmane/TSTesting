# AI Tagging — Fields and Approach

---

## Fields in scope for AI auto-tagging

| Field | Type | AI source |
|---|---|---|
| **Subject Matter** | Multi-select | OpenAI classification |
| **Contract Type** | Single-select | OpenAI classification |
| **Term Category** | Single-select | OpenAI reasoning |
| **Effective Date** | Date | OpenAI reasoning |
| **End Date** | Date | OpenAI reasoning |
| **Action Date Notice Period** | Text | OpenAI extraction |
| **Contract Description/Notes** | Free text | OpenAI summarization |

**Out of scope for AI:** Owning Business Unit and Business Owners (autofilled from request creation), Status (workflow-managed), Signed (boolean).

---

## Short term — Analyst-triggered AI Assist

A new **AI Assist** button is added to the existing Request form. The analyst clicks it when ready to tag; the form pre-populates with AI suggestions plus confidence indicators and rationale tooltips; analyst reviews, edits, saves. Existing workflow is preserved as a fallback if AI fails.

### Flow

1. Analyst clicks **AI Assist** in the form
2. SPFx calls Azure Function with request and document IDs
3. Function fetches the PDF from SharePoint
4. Function calls **Document Intelligence (`prebuilt-contract`)** → structured facts (parties, execution date, jurisdictions)
5. Function calls **Azure OpenAI** in two passes:
   - **Pass 1** classifies Contract Type and Subject Matter
   - **Pass 2** uses a type-specific prompt to extract Term Category, dates, notice period, description
6. Form renders suggestions with confidence badges (High/Medium/Low) and rationale tooltips
7. Analyst reviews, edits, saves. Both AI suggestion and final value are stored for audit and accuracy measurement.

### Why Document Intelligence first

- **Prebuilt by Microsoft, no training needed** — accurate on contract facts (parties, dates, jurisdictions) out of the box
- **Handles all three template scenarios** — looks for contract concepts, not fixed layouts
- **Faster and cheaper than OpenAI** for what it does — establishes a reliable baseline before interpretation
- **Schema is fixed, no reasoning** — doesn't extract business-specific fields like Term Category or compute Effective Date from clauses, which is why OpenAI runs after

### Why OpenAI in two passes

- **Covers what DI can't** — Term Category, computed dates (e.g., "effective 90 days after execution"), notice periods, summaries
- **DI output anchors OpenAI** — structured input reduces hallucination risk
- **Type-specific prompts are more accurate** — an NDA prompt knows what NDAs have and don't have; a one-prompt-fits-all is necessarily weaker
- **Pass 1 routes to Pass 2** — once Contract Type is detected, the right prompt template runs for extraction
- **Cost of the extra call is minimal vs the accuracy gain**

## Long term — Background processing + selective auto-acceptance

Same architecture, different trigger and level of automation.

**Phase 2 — Background processing.** A SharePoint webhook fires when a contract is executed, drops a message on a queue, the same Function processes it asynchronously, Power Automate notifies the analyst when suggestions are ready. Analyst becomes a reviewer rather than a starter.

**Phase 3 — Selective auto-acceptance.** For high-confidence fields on contracts using our template, auto-accept the AI value without analyst review. Analysts only see contracts that need attention — vendor templates, low-confidence fields, edge cases. Shifts the model from "AI assists analyst" to "AI handles routine, analyst handles exceptions."

### Why this for the long term

- **Suggestions ready before analyst opens the contract** (Phase 2)
- **Auto-acceptance only enabled where accuracy is proven** (Phase 3) — confidence threshold + historical accuracy data from Phase 1 gate which fields can auto-accept
- **Reuses Phase 1 infrastructure** — same Function, same AI services, only the trigger changes
- **Compounding value** — Phase 1 reduces *time per contract*; Phase 3 reduces *number of contracts needing human attention*

---

## Why phased, not all at once

- **De-risks** — each phase validates the next; accuracy proven before automation removes the human
- **Builds analyst trust** — automation comes after suggestions have earned credibility
- **Data-driven** — Phase 3 auto-acceptance is justified by real accuracy data, not estimates
- **Existing workflow preserved** — manual tagging remains functional; AI failures don't block the business

## Summary

| | Short term | Long term |
|---|---|---|
| **Trigger** | Analyst clicks "AI Assist" | Automatic on contract execution |
| **Analyst role** | Reviews every contract | Reviews exceptions only |
| **Automation** | Human-in-the-loop | Selective auto-acceptance |
| **Business risk** | None — analyst confirms everything | Low — only auto-accept where accuracy is proven |
| **Primary value** | Reduces time per contract | Reduces contracts needing human attention |
